## Findings

### Finding 1 — Daily-salt for IP/UA hashing is public-derivable [HIGH]

* **Severity:** HIGH
* **Confidence:** 9/10
* **Status:** VERIFIED (code-traced)
* **Phase:** 11 — Data Classification / OWASP A02
* **Category:** Cryptographic Failure / Privacy
* **File:** `src/lib/geo.ts:42-56`

```ts
export function hashIp(ip: string | null | undefined): string | null {
  if (!ip) return null;
  const dailySalt = new Date().toISOString().slice(0, 10);  // ← public
  return createHash('sha256').update(`${dailySalt}::${ip}`).digest('hex');
}
```

The "daily salt" is `YYYY-MM-DD` — public, derivable. SHA-256 over IPv4 keyspace (2^32) brute-forces in ~5 min on commodity GPUs (~10B SHA-256/sec). Same applies to `hashUserAgent`, though UA is higher-entropy and harder to enumerate.

**Exploit scenario:**
1. Attacker gains read access to `consent_audit.ip_hash` (DB compromise, RLS misconfiguration, insider threat, subpoena to Supabase).
2. For each row, brute-force the IPv4 keyspace using the row's known date as salt → ~5 min/day.
3. Reverse-map ip_hash → IP address.
4. Cross-reference IP via geo-IP / ISP records → identify users.
5. Result: "User from IP X.X.X.X consented to **political-data sale** on date Y" — re-identification of users to political opinions, classified as sensitive data under CO/CT/VA/UT privacy statutes.

**Impact:** Defeats the stated PII-minimization design. Plan §14.3 promised "ip_hash: SHA-256(ip || daily_salt), not raw IP" — implementation makes the salt public and the hash effectively reversible. Reputational + legal risk in jurisdictions treating political views as sensitive.

**Recommendation:**
- Move salt to a server-side secret. Add `IP_HASH_SECRET` to env (rotated monthly), keep out of logs and responses. HMAC-SHA-256(secret, ip) in place of plain SHA-256.
- OR truncate hash to 8-12 hex chars + store coarse geo separately. Accept some collisions in exchange for true irreversibility.
- Do NOT keep a deterministic public salt of any form — every variant fails the same brute-force attack.

### Finding 2 — `/api/match` has no rate limiting; LLM cost amplification [HIGH]

* **Severity:** HIGH
* **Confidence:** 9/10
* **Status:** VERIFIED (code-traced)
* **Phase:** 7 — LLM Security
* **Category:** LLM Cost / Resource Exhaustion (financial risk, NOT DoS — explicit Phase 12 exception)
* **File:** `src/app/api/match/route.ts:25-30`, `src/lib/llm/match.ts:90-121`

```ts
* TODO (Chunk 5/6):
*  - Rate-limit by session_id (10/hr) and IP (30/hr) via cookie + middleware
```

Plan §6 specified per-session and per-IP rate limits and a $50/day spend cap. Neither is implemented. The in-memory cache is keyed on `(race_id, free_text)` so any unique free-text bypasses it. The MATCH_API_DISABLED env var is a reactive kill switch, not a preventive control.

**Exploit scenario:**
1. Attacker scripts POST `/api/match` with 1 unique random suffix per request → cache miss every time.
2. Each call → Haiku (~$0.001 + ~1-2s).
3. 60 req/min × 1 hr = 3,600 calls × $0.001 = ~$3.60/hr per IP. Distributed across 10 IPs: ~$864/day.
4. With Anthropic's standard rate limits + your dashboard cap absent, weekly bill from a single attacker can exceed your project budget.

**Impact:** Direct financial exposure once `ANTHROPIC_API_KEY` is set in production. The plan called this out as a key cost-control requirement.

**Recommendation:** Before enabling Anthropic in prod, ship at least one of:
- Edge middleware rate-limiter keyed on session+IP (Vercel KV / Upstash Redis token bucket).
- Persist `llm_matches` cache to Supabase as planned (cross-session cache hits, not just same-instance).
- Anthropic dashboard hard daily spend cap (defense in depth, not primary control).
- Optional: require completed quick-poll (>=3 weights) before allowing free-text — raises attacker effort.
